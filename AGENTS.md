# AGENTS.md - 项目开发规则

> 本项目所有代码应遵循以下设计原则

## 架构原则

### 1. 模块化、组件化、可插拔设计

- **模块化 (Modularity)**: 功能拆分为独立的模块，每个模块有清晰的单一职责
- **组件化 (Component-based)**: UI 和逻辑都采用组件化思维，组件之间松耦合
- **可插拔 (Pluggability)**: 核心功能通过接口/抽象定义，具体实现可替换、可扩展
- **扩展性 (Extensibility)**: 设计时考虑未来需求，预留扩展点，避免硬编码

**实践要点**:
- 使用接口/抽象类定义契约，而非直接依赖具体实现
- 避免循环依赖，模块间通过明确的 API/已经定义好的规则 通信
- 配置与代码分离，关键行为应可配置
- 新增功能不应要求修改现有核心代码（开闭原则）

---

## 2. 代码质量与规范

- **代码风格统一**：使用 lint/format 工具强制统一风格，避免风格争议
- **复杂度控制**：单个函数不超过 50 行，圈复杂度不超过 10
- **命名规范**：语义化命名，禁止使用无意义缩写（如 `tmp`、`x`），命名即文档

## 3. 测试策略

- **核心模块必须有单元测试**：测试用例覆盖正常路径和边界情况
- **新功能伴随测试提交**：不接受无测试的新功能 PR
- **覆盖率要求**：核心代码测试覆盖率不低于 80%

## 4. 文档与注释

- **公共 API 必须有文档注释**：说明用途、参数、返回值、异常
- **复杂逻辑注释"为什么"**：代码本身说明"做什么"，注释解释"为什么"
- **文档同步更新**：修改代码时同步更新相关文档，禁止文档与代码脱节

## 5. 版本控制

- **提交信息规范**：遵循 Conventional Commits（`feat:`、`fix:`、`docs:`、`refactor:` 等）
- **分支策略**：
  - `main`：稳定分支，始终可运行
  - `develop`：开发分支，功能合并目标
  - `feature/*`：功能分支，从 develop 切出，完成后 PR 回 develop

## 6. 安全与性能

- **输入验证**：不信任任何外部输入，必须校验和消毒
- **关键操作留痕**：敏感操作必须有日志记录，便于审计
- **避免过早优化**：第一版优先正确性，核心路径预留优化空间

## 7. 依赖管理

- **引入依赖需评估**：评估维护状态、体积、许可证，不引入不必要的依赖
- **优先使用标准库**：能用标准库解决的，不引入第三方依赖
- **避免重复引入**：检查现有依赖是否已提供所需功能

---

*规则随项目演进可调整，修改需经讨论*
